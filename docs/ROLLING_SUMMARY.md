# Rolling Summary

How March compresses and deduplicates conversation history into a compact rolling summary.

---

## Overview

The **rolling summary** is a compressed representation of all past conversation context. It replaces old messages after compaction, allowing the LLM to maintain continuity across long sessions without exceeding the context window.

The rolling summary is stored in `session.rolling_summary` and persisted in the `sessions` table in SQLite.

---

## Generation: Two-Step Process

Rolling summaries are generated by `Agent._two_step_compaction()`, which runs two LLM calls:

### Step 1 — Summarize

Compress the existing rolling summary + all current messages into a single conversation summary.

**Input:** Previous rolling summary (if any) + full message history  
**Prompt instructions:**
- Keep decisions, rationale, action items, file paths, URLs, commands, code snippets
- Drop greetings, failed debugging attempts, redundant explanations, verbose tool output
- Write concise narrative with bullet points — not a transcript
- Target: ≤30% of input size

**Output:** A compressed conversation summary

### Step 2 — Dedup Against Static Files

Remove content from the summary that duplicates information already present in the system's static `.md` files and session memory.

**Input:**
- System rules (`SYSTEM.md`)
- Agent profile (`AGENT.md`)
- Tool inventory (`TOOLS.md`)
- Long-term memory (`MEMORY.md`)
- Session facts (`facts.md`)
- Session plan (`plan.md`)
- The conversation summary from Step 1

**Prompt instructions:**
- Remove anything that duplicates the static files
- Keep all unique conversation-specific details, decisions, and facts
- Keep session facts and plans verbatim
- Preserve all identifiers in backticks
- Target size: `min(dedup_max_ratio × context_window, current_token_count)` tokens

**Output:** A self-contained rolling context document — the new `rolling_summary`

```
Step 1                              Step 2
┌──────────────┐                    ┌──────────────────────┐
│ rolling_sum  │                    │ SYSTEM.md            │
│ + messages   │─── Summarize ───►  │ AGENT.md             │
│              │    (LLM call)      │ TOOLS.md             │
└──────────────┘         │          │ MEMORY.md            │
                         │          │ facts.md / plan.md   │
                         ▼          │ + Summary from Step 1│
                  ┌──────────┐      └──────────┬───────────┘
                  │ Summary  │                  │
                  └──────────┘      Dedup (LLM call)
                                                │
                                                ▼
                                    ┌──────────────────────┐
                                    │  New Rolling Summary  │
                                    │  (self-contained)     │
                                    └──────────────────────┘
```

## How the Rolling Summary Is Used

### As System Prompt

After the first compaction, the rolling summary **becomes the system prompt**. The agent's `_build_context()` method checks:

```python
if session.rolling_summary:
    # Use rolling summary as system_rules — skip reading .md files
    return Context(system_rules=session.rolling_summary, ...)
else:
    # First turn — read all .md files
    return Context(
        system_rules=load("SYSTEM.md"),
        agent_profile=load("AGENT.md"),
        tool_inventory=load("TOOLS.md"),
        long_term_memory=load("MEMORY.md"),
    )
```

This works because the dedup step in Step 2 already folded the essential parts of the `.md` files into the rolling summary (while removing duplicates). The result is a single, compact system prompt that contains everything the LLM needs.

### Prepended to Messages

When building the message list for the LLM, the rolling summary is prepended as a `user` message:

```json
{
  "role": "user",
  "content": "[Context Summary — rolling context from previous compaction]\n\n{rolling_summary}"
}
```

This appears before any recent messages, giving the LLM the historical context it needs.

---

## Fallback Behavior

- **Step 1 returns empty:** Falls back to the existing `rolling_summary` (no change)
- **Step 2 returns empty:** Falls back to the Step 1 summary (no dedup, but still compressed)
- **LLM call fails entirely:** Falls back to simple truncation (drop oldest messages, keep recent ones)

---

## Session Memory Folding

During compaction (in `compact_messages()`), session memory is also processed:

1. **Load** `facts.md` and `plan.md` from `~/.march/memory/{session_id}/`
2. **Dedup** the session memory files themselves (via `dedup_session_memory()`)
   - Removes true duplicates and superseded entries
   - Preserves all unique details — this is dedup, not compression
   - Writes cleaned files back to disk
3. **Fold** the cleaned session memory into the rolling summary as a `**Session Memory:**` section

This ensures session memory survives compaction and remains visible to the LLM even after old messages are discarded.

## Size Targets

| Parameter | Default | Purpose |
|-----------|---------|---------|
| `summary_budget_ratio` | 0.15 | Fraction of context window reserved for the summary |
| `dedup_max_ratio` | 0.30 | Max fraction of context window for the dedup output |
| Summarize target | ≤30% of input | Compression ratio for Step 1 |
| Dedup target | `min(30% × context_window, current_tokens)` | Size cap for Step 2 output |

For a 200K context window:
- Summary budget: ~30K tokens
- Dedup target: ~60K tokens max

---

## Key Properties

1. **Self-contained** — After dedup, the rolling summary contains everything the LLM needs without re-reading `.md` files.
2. **Lossless for important data** — Decisions, identifiers, code, and action items are preserved. Only filler and redundancy are dropped.
3. **Accumulative** — Each compaction builds on the previous rolling summary, so context from early in the session is never fully lost — just increasingly compressed.
4. **Idempotent session memory** — Session memory files accumulate across compactions and are deduplicated each time, preventing unbounded growth while preserving all unique information.
